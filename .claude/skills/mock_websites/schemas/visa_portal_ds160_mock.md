# visa_portal_ds160_mock Schema

**Deploy order**: 54 (alphabetical among all *_mock dirs, BASE_PORT=8000 -> port 8054)
**Base URL**: `http://172.17.46.46:8054/`
**Go Endpoint**: `GET /go?sid=<sid>` -> `{initial_state, current_state, state_diff}`
**Inject**: `POST /post?sid=<sid>` with body `{"action":"set","state":{...}}` (use `"merge":true` for partial update)
**Update current only**: `POST /post?sid=<sid>` with body `{"action":"set_current","state":{...}}`
**Reset**: `POST /post?sid=<sid>` with body `{"action":"reset"}`
**Upload files**: `POST /upload?sid=<sid>` (multipart/form-data) -> `{success, files: [{original_name, stored_name, size, content_type, url}]}`
**Serve files**: `GET /files/<sid>/<filename>`

Note: vite.config.js uses `port: 0` (random). Actual port depends on deployment config. The inject endpoint is `/post`, not `/go`. State is stored in localStorage under keys `ds160_state` / `ds160_initialState` (or `ds160_state_<sid>` / `ds160_initialState_<sid>` when a session ID is provided).

## Application Overview

This mock simulates the **U.S. Nonimmigrant Visa Application (DS-160)** form from the Consular Electronic Application Center. It is a multi-step form wizard that walks through sections of the visa application: personal information, address/phone, passport, travel, family, work/education, security, photo upload, review, and sign/submit.

### Routes

| Route | Component | Description |
|-------|-----------|-------------|

| `/retrieve` | `RetrieveApplication` | Retrieve a previously saved application by ID and security question |

| `/application/id-generation` | `AppIdDisplay` | Displays the generated Application ID; user records it before continuing |

| `/application/personal2` | PlaceholderPage | Personal Information 2 (placeholder) |
| `/application/address` | PlaceholderPage | Address & Phone (placeholder) |
| `/application/passport` | PlaceholderPage | Passport (placeholder) |
| `/application/travel` | PlaceholderPage | Travel (placeholder) |
| `/application/usContact` | PlaceholderPage | U.S. Contact (placeholder) |
| `/application/family` | PlaceholderPage | Family (placeholder) |
| `/application/work` | PlaceholderPage | Work / Education / Training (placeholder) |
| `/application/security` | PlaceholderPage | Security and Background (placeholder) |
| `/application/photo` | PlaceholderPage | Photo (placeholder) |

Routes under `/application/*` (except `security-question` and `id-generation`) are rendered inside `ApplicationLayout`, which provides a top info bar (location, application ID, session timer), a progress bar, and a sidebar navigation panel.

## State Schema

The entire application state is a single flat object with these top-level keys:

| Key | Type | Description |
|-----|------|-------------|
| `ds160Application` | object | Application metadata: ID, location, status, timestamps, current section |
| `personalInfo` | object | Contains `personal1` and `personal2` sub-objects for applicant personal details |
| `addressAndPhone` | object | Home/mailing address, phone numbers, email addresses |
| `meta` | object | Session metadata: last saved time, session expiration |

### `ds160Application` subfields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `applicationId` | string | `""` | Auto-generated ID in format `AA00XXXXXX` (4 prefix chars + 6 random alphanumeric). Generated by `generateAppId()` when starting a new application. |
| `location` | string | `""` | Embassy/consulate location code. Possible values: `"CAN"` (Canada, Toronto), `"CHN"` (China, Beijing), `"FRA"` (France, Paris), `"GBR"` (United Kingdom, London), `"IND"` (India, New Delhi), `"MEX"` (Mexico, Mexico City) |
| `status` | string | `"Not Started"` | Application status. Known value: `"Not Started"` |
| `createdAt` | string\|null | `null` | ISO 8601 timestamp set when application is created (when `generateAppId()` runs) |
| `updatedAt` | string\|null | `null` | ISO 8601 timestamp (not currently set by any handler in the code) |
| `currentSection` | string | `"start"` | Tracks current form section. Possible values: `"start"`, `"personal1"`, `"personal2"`, `"address"`, `"passport"`, `"travel"`, `"travelCompanions"`, `"previousTravel"`, `"usContact"`, `"family"`, `"work"`, `"security"`, `"photo"`, `"review"`, `"sign"` |

## State Management Details

- **Pattern**: React Context (`AppContext.jsx`)
- **Context provider**: `AppProvider` wraps entire app
- **State update**: `updateState(path, value)` -- uses dot-notation path (e.g., `"personalInfo.personal1.surname"`)
- **Persistence**: localStorage with keys `ds160_state` (or `ds160_state_<sid>`) and `ds160_initialState` (or `ds160_initialState_<sid>`)
- **Session ID**: Read from URL query param `?sid=<value>` or from `sessionStorage` key `mock_sid`
- **Custom state injection**: On first load for a session, fetches `GET /state?sid=<sid>` to check for server-injected state. Deep-merges with defaults.
- **Initial state snapshot**: Captured once on first load; used by `/go` for diff calculation

## Observable State Changes (for LLM evaluation)

| User Action | State Field(s) Changed |
|-------------|------------------------|
| Select embassy location and click "START AN APPLICATION" | `ds160Application.location` set to location code (e.g. `"CHN"`), `ds160Application.applicationId` set to generated ID (e.g. `"AA00K7M2P9"`), `ds160Application.createdAt` set to ISO timestamp |
| Fill in surname on Personal 1 page | `personalInfo.personal1.surname` set to entered value |
| Fill in given name on Personal 1 page | `personalInfo.personal1.givenName` set to entered value |
| Enter native full name | `personalInfo.personal1.nativeFullName` set to entered value |
| Check "Does Not Apply" for native name | `personalInfo.personal1.nativeFullNameDoesNotApply` -> `true` |
| Select "Yes" for other names used | `personalInfo.personal1.otherNamesUsed` -> `"Yes"` |
| Check "Does Not Apply" for state of birth | `personalInfo.personal1.pobStateDoesNotApply` -> `true` |
| Retrieve application with matching ID | `ds160Application.applicationId` and `ds160Application.location` updated from form input; navigates to `/application/personal1` |

## Notes

- Only `PersonalInfo1` has full form field binding to state. Other sections (`personal2` through `sign`) are placeholder pages with static content and "Next Section" navigation links.
- The `SecurityQuestion` page does NOT persist the chosen question or answer to state -- it only navigates forward on submit.
- The `RetrieveApplication` page validates the entered Application ID against `state.ds160Application.applicationId` in localStorage. If they do not match, a red error banner is shown.
- The `generateAppId()` function creates IDs with prefix `"AA00"` followed by 6 random characters from `A-Z0-9`.
- The comment `// ... other sections initialized empty ...` in `AppContext.jsx` suggests the INITIAL_STATE in the source code may be truncated; the actual default state includes `ds160Application`, `personalInfo`, `addressAndPhone`, and `meta`.
